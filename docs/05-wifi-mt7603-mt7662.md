## Summary

The Archer XR500v provides dual-band WiFi through **two separate MediaTek PCIe radios**, not a single dual-band chip: a **MT7603** (2T2R 802.11b/g/n) on PCIe port0 for 2.4 GHz, and a **MT7662 / MT76x2** (2T2R 802.11a/n/ac) on PCIe port1 for 5 GHz. Both are brought up under the mainline **mt76** driver stack (`mt76x2e` for the MT7662, `mt7603e` for the MT7603). The 5 GHz radio came up early once the PCIe enumeration, EEPROM/nvmem mapping, and `wpad` were in place. The 2.4 GHz radio required two additional pieces: its PCIe endpoint did not respond to configuration transactions until the OEM's *global* PCIe/SCU reset sequence was replicated at boot, and once enumerated the `mt7603e` driver hung while uploading a blank EEPROM to the chip MCU — addressed with a synthetic EEPROM inlined in the device tree. Both items are resolved, with one operational requirement noted: **the MT7603 enumerates only on a cold boot** (a full power cycle), because the SCU reset/clock writes that gate enumeration are applied during the PCIe driver probe and do not re-run on a warm reboot or a `bflag`-driven boot. Under mainline this device maps **phy0 = MT7603 (2.4 GHz)** and **phy1 = MT7662 (5 GHz)**, and both radios run as APs concurrently. Two operational issues are documented below: a kernel panic under sustained WiFi-to-LAN forwarding (an `rx-gro-list` / `NETIF_F_GRO_FRAGLIST` interaction with the DSA forward path) and an MT7603 MCU timeout that can wedge the radio after long uptime.

This page documents the owner's own device. The bring-up was reverse-engineered against the OEM Linux 2.6.36/3.18 source and the running stock firmware; relevant exact addresses and values are given where they were verified.

## PCIe topology

The two radios sit behind two independent EN7528-class PCIe root complexes driven by the mainline `pcie-mediatek.c` controller (`compatible = "econet,en7528-pcie"`). See [02-hardware-chip-inventory.md](02-hardware-chip-inventory.md) for the controller itself; the WiFi-relevant facts:

| Radio | Chip ID | Controller | DT domain | Slot node | Linux BDF | Driver | phyN |
|-------|---------|-----------|-----------|-----------|-----------|--------|------|
| 2.4 GHz | `14c3:7603` | port0 @ `0x1fb81000` | `linux,pci-domain = <0>` | `slot0: pcie@0,0` (`reg = <0x0000 …>`) | `0000:01:00.0` | `mt7603e` | phy0 |
| 5 GHz | `14c3:7662` | port1 @ `0x1fb83000` | `linux,pci-domain = <1>` | `slot1: pcie@1,0` (`reg = <0x0800 …>`) | `0001:01:00.0` | `mt76x2e` | phy1 |

A key insight from the 5 GHz bring-up was the **PCI BDF / devfn decoding**. The MT7662's root-port slot is `pcie@1,0` with `reg = <0x0800 …>`, i.e. **device 1, function 0** (`devfn = 8`), not "function 1". Early attempts placed the `wifi@0,0` child under a wrong-devfn node, leaving the WiFi node orphaned (a sibling rather than a child of the root port), so `mt76` never bound to a device with a valid `of_node`. The fix was to attach the WiFi child via the base dtsi's existing `&slot1` phandle. The diagnostic that revealed this was checking `/sys/bus/pci/devices/<BDF>/of_node` — when absent or pointing to the wrong node, the OF mapping was broken.

A second prerequisite, common to both radios, is the `pciecfg` syscon node. The `mtk-pcie` driver looks up `of_find_compatible_node(... "mediatek,generic-pciecfg")`; without it the driver bails with `EN7528: pciecfg syscon not available` and **both** ports report link down, so neither radio enumerates:

```dts
pciecfg: pciecfg@1fb80000 {
    compatible = "mediatek,generic-pciecfg", "syscon";
    reg = <0x1fb80000 0x1000>;
};
```

The PCIe controller in turn depends on the `chipscu` syscon (`econet,en751221-chip-scu`) and `CONFIG_MFD_SYSCON=y` for the EN7523 clock driver to resolve `sys_ck0`/`sys_ck1`, otherwise `clk-en7523` returns `EPROBE_DEFER` indefinitely and the PCIe ports never probe.

## 5 GHz: MT7662 / MT76x2 (mt76x2e)

The 5 GHz radio is a MediaTek MT7662 (reported ASIC revision `76120044`), driven by `mt76x2e`. It enumerates as `0001:01:00.0`. Bring-up requirements:

1. **`kmod-mt76x2`** in `DEVICE_PACKAGES` (and `CONFIG_PACKAGE_kmod-mt76x2=m` in `.config` — it is not pulled in automatically).
2. **`wpad-basic-mbedtls`** for `hostapd`; without it `wifi up` does nothing (no `hostapd`, empty `iw dev`).
3. **Factory EEPROM via nvmem.** The MT7662 calibration data (per-channel TX power, etc.) is stored in the NAND **`misc` partition at offset `0xE0000`** (512 bytes, chip-ID `0x7662` confirmed in byte 0). It is wired in the DTS as an nvmem cell and consumed by the driver. Without it, the driver falls back to conservative defaults and TX power drops to ~3 dBm. With it, the radio reaches **20 dBm** and runs **VHT80 (802.11ac)**.
4. **MAC from flash.** The factory MAC `5c:a6:e6:b2:04:38` lives in `misc` at offset `0xF100`.

The WiFi node lives under the port1 slot:

```dts
&slot1 {
    wifi@0,0 {
        /* MT7662 dual-band 802.11ac */
        compatible = "mediatek,mt76";
        reg = <0x0000 0 0 0 0>;
        nvmem-cells = <&eeprom_misc_e0000>, <&macaddr_misc_f100>;
        nvmem-cell-names = "eeprom", "mac-address";
        led {
            led-sources = <2>;     /* MT7662 MAC LED pin 2, active-low */
            led-active-low;
        };
    };
};
```

The 5 GHz panel LED is driven directly by the MT7662's MAC LED block (`MT_LED_CTRL`, pin 2, active-low); `mt76` auto-assigns the `phy1tpt` throughput trigger, so no `board.d`/uci LED wiring is needed. See [08-front-panel-leds.md](08-front-panel-leds.md).

### Note on the MT7662 and the 2.4 GHz band

The MT7662 silicon advertises 2.4 GHz capability (channels 1–13, 20 dBm appear in `iw phy`). It is a single radio, so it can only operate one band at a time, and the board's 2.4 GHz RF front end (PA/LNA/antenna) is not wired to this chip — it is wired for 5 GHz. Forcing `radio` to `band=2g` produced a clean `AP-ENABLED` and beaconing at 2437 MHz, but no client could associate: the chip transmits but the 2.4 GHz path does not radiate on this board. The real 2.4 GHz comes from the second chip, the MT7603. (Same "capability in silicon, board wiring decides" pattern as the WiFi LEDs.)

## 2.4 GHz: MT7603 (mt7603e)

The MT7603 (ASIC revision `76030010`) is a *second*, separate PCIe radio on port0, which the OEM uses for all 2.4 GHz traffic. It is brought up under mainline `mt7603e`. Two distinct problems had to be solved: the PCIe endpoint did not enumerate, and once it did the driver hung uploading a blank EEPROM. Both are resolved.

### Enumeration — the OEM global reset

Under the stock `en7528` per-port reset, the MT7603 endpoint **trained the PCIe link at the PHY level (width 1, 2.5 GT/s) but returned UR (Unsupported Request) to every configuration TLP**, so it never enumerated. This matched the known, unresolved upstream EcoNet observation that one card (MT7662) works while the other (MT7603) does not.

An extensive elimination pass ruled out the obvious causes: PHY analog tuning (already correct in mainline), GPIO power/PERST, SCU clock enable, MAC reset bits, CRS retry timing, settle delays, reset-retry loops, BAR0 setup, and the config-TLP "tag" field. Several intermediate findings were real but insufficient on their own:

- The port0 link was **flapping** (`0x4 ↔ 0x6` in `EN7528_LINKUP_REG`) because `PCIE_LINKDOWN_RST_EN` re-asserted on every marginal glitch. Clearing that bit after link-up gave a **stable** link (steady `0x6`), but config reads still returned UR.
- The OEM config-read path uses **CFG_HEADER1 tag `0x070f`** (tag `0x07` in bits 8–10 + byte-enables `0x0f`); mainline wrote only `0x000f`. With the tag, the completion-status field changed from `0xe0` (UR) to non-UR — but the endpoint config space still read as zero.
- The config engine could wedge (`APP_CFG_REQ` stuck); writing `RST_CTRL 0x103 -> 0x10f` (assert then deassert MAC+CRSTB) cleared it.

None of those, alone or combined, made the endpoint respond. The key insight came from reading the OEM kernel's `mt7512_pcie_reset()` / `mt7512_pcie_set_mac()` in `arch/mips/pci/pci-tc3162u.c`: the OEM performs a **global PCIe reset sequence** at boot that the mainline `en7528` startup path omits entirely. Replicating it inside `mtk_pcie_startup_port_en7528()` for `slot == 0` makes the endpoint enumerate. The sequence (from patch `913-mt7603-port0-enable.patch`):

```
# SCU global reset (executed during the driver probe, where SCU writes are safe)
0x1fb00084 |= 0x2                             # SCU clock gate for port0 (bit 1)
0x1fb00088 |= BIT(22)                         # enable port ; sleep 1 ms
0x1fb00834 &= ~((1<<26)|(1<<27)|(1<<29))      # controller reset: assert ; sleep 1 ms
0x1fb00834 |=  ((1<<26)|(1<<27)|(1<<29))      #                  release ; sleep 100 ms
0x1fb00834 &= ~((1<<26)|(1<<27)|(1<<29))      #                  assert  ; sleep 5 ms
0x1fb00088 &= ~((1<<29)|(1<<26))              # device reset: assert ; sleep 1 ms
0x1fb00088 |=  ((1<<29)|(1<<26))              #              release
msleep(250)                                   # settle
# then per-port reset + link training, then MAC setup:
0x1fb8111c &= ~BIT(5)                          # MSI off
0x1fb81420 &= ~BIT(16)                         # INTx on
pciecfg 0x40 |= 0x03                           # error interrupts
# plus: keep PCIE_LINKDOWN_RST_EN clear, CFG_HEADER1 tag 0x070f, exact APP_CFG_REQ write
```

**Timing constraint:** writes to the SCU region `0x1fb000xx` **crash the kernel at runtime** (they glitch SoC clocks) but are **safe during the PCIe driver probe** at boot, because the rest of the system is not yet up. This is why every runtime poke of the reset registers wedged the device, while the same writes from inside `startup_port_en7528()` succeed. With this in place, `pci 0000:01:00.0: [14c3:7603] PCIe Endpoint` enumerates and a BAR is assigned.

**Cold boot is required.** Because the SCU reset/clock writes only run during the PCIe probe at power-on, the MT7603 enumerates reliably only on a full **cold boot** (power cycle). A warm reboot, or a boot driven via the bootloader `bflag` slot switch, does not re-apply the SCU sequence, so the endpoint stays in the non-enumerating state until the next power cycle. See [03-boot-partitions-flashing.md](03-boot-partitions-flashing.md) for the `bflag`/boot-slot mechanics.

> Caveat: patch 913 still carries diagnostic scaffolding — `dev_info` logging, an `en7528_mt7603_cfg_ready()` probe, the GPIO16/27 + SCU-`0x84` clock pokes, and four port0 PHY analog deltas in `phy-en7528-pcie.c`. The PHY deltas in particular are believed unnecessary and are flagged for bisection. The committed patch is functional; the minimal subset has not yet been isolated.

### The MCU EEPROM hang and the synthetic EEPROM

Once enumerated, `mt7603e` loaded, reported `Invalid MAC address, using random address`, ran firmware init, and then **hung the kernel inside `mt7603_mcu_set_eeprom`** (instrumentation confirmed markers up to "post dma_sched_init" then a hang before the next marker). The driver was pushing a **blank EEPROM** to the chip MCU (`MCU_EXT_CMD_EFUSE_BUFFER_MODE`) with an all-`0xff` TX table.

The reason there was no real EEPROM: investigation of the flash and the chip efuse showed there is **no factory MT7603 EEPROM anywhere**:

- The on-chip **efuse is 960/1024 bytes blank** (`0xff`). It contains *partial* RF cal — XTAL freq offset (`0xf4 = 0xa4`), XTAL RFCAL (`0xf7 = 0x88`), temp cal (`0x55 = b8 40 c8`), a 2G TX-power start value (`0x56/0x5c = 40 c8`), FT version (`0xf0 = 05`) — but **no MAC, no chip-ID, no full TX table, no NIC config**. (Efuse dumped from stock via the MT7603 BAR at `0x20000000`, window `BAR+0xF0000`, mt76 sequence; confirmed no byteswap by reading chip-id `0x7603` at `BAR+0x1008`.)
- The NAND `misc` partition contains the MT7662 EEPROM at `0xE0000` but **no MT7603 EEPROM block** — the OEM driver `mt7603eap.ko` ships a built-in default instead.

The fix is a **synthetic 1024-byte EEPROM inlined into the device tree** via `mediatek,eeprom-data` (mt76 supports an inline EEPROM blob, so no flash write is needed). It carries chip-ID `0x7603`, a **placeholder MAC**, `NIC_CONF = 0x22` (2T2R), the **real partial cal copied out of the efuse** (XTAL/temp/TX-power-start), and conservative TX-power defaults for the values the efuse lacked. With a valid EEPROM, `mt7603_mcu_set_eeprom` completes and the driver finishes registration.

```dts
&slot0 {
    wifi@0,0 {
        compatible = "mediatek,mt76";
        reg = <0x0000 0 0 0 0>;
        nvmem-cells = <&macaddr_misc_f100>;
        nvmem-cell-names = "mac-address";
        mac-address-increment = <1>;
        mediatek,gpio-led;          /* 2.4 GHz panel LED, see below */
        mediatek,eeprom-data = /bits/ 8 <
            0x03 0x76 0x00 0x00 ... /* 1024 bytes, generated by make_mt7603_eeprom.py */ >;
    };
};
```

The first bytes encode chip-ID `0x03 0x76` (little-endian `0x7603`) and a placeholder MAC `02:76:03:50:00:01` at offset 4; `NIC_CONF = 0x22` sits at offset `0x34`; the efuse-derived cal (`b8 40 c8`, `40 c8`, FT `05`, `a4`, `88`) is placed at the matching offsets. Only the TX-power table is approximated with conservative defaults (the efuse never held a complete one).

This synthetic EEPROM uses a non-factory placeholder MAC and a conservative TX-power table. The placeholder MAC is then **overridden at runtime by the nvmem MAC** described below, so the address that actually appears on air is the factory-derived one — but the per-channel TX-power calibration remains the conservative synthetic table, not factory data. (For contrast, the 5 GHz MT7662 reaches its rated 20 dBm from a *real factory EEPROM* in flash; the 2.4 GHz MT7603 has no such factory EEPROM, only partial efuse cal.)

### MAC address derivation

The 2.4 GHz radio uses the **factory MAC base** in `misc @0xF100` plus one, yielding a per-device, universally-administered MAC (`5c:a6:e6:b2:04:39`, vs. eth/5 GHz `…:04:38`). This needs `mac-address-increment` support in `of_get_mac_address()`, which the cjdelisle econet 6.12 tree lacked; it is added by kernel patch `700-of-net-mac-address-increment.patch` (reads `mac-address-increment` and optional `mac-address-increment-byte`, default byte index 5, and adds the increment to the base MAC). The synthetic EEPROM's placeholder MAC is overridden by this nvmem MAC, so the real factory-derived address is what actually appears on air (`phy0-ap0 = 5c:a6:e6:b2:04:39`, verified live).

### Build/packaging

```
DEVICE_PACKAGES := kmod-mt76x2 kmod-mt7603 kmod-usb-ledtrig-usbport \
                   wpad-basic-mbedtls luci kmod-econet-pcm
```

`kmod-mt7603` and `kmod-mt76x2` must also be enabled in `.config`. The `913-mt7603-port0-enable.patch` (PCIe reset, in `target/linux/econet/patches-6.12/`) and the inline DTS EEPROM together constitute the 2.4 GHz recipe.

### 2.4 GHz panel LED

The MT7603's 2.4 GHz panel LED is wired to the chip's **internal GPIO12 + GPIO13** (both drive the same LED, active-low), reached only through the PCIe remap-2 window via `mt7603_reg_map()`. Mainline mt76's MT7603 LED path targets the MAC LED pad (`0x80024000`), which is not routed to this board's LED. An mt76 patch (`004-mt7603-xr500v-chip-gpio-led.patch`) adds a `mediatek,gpio-led` DT property; when present, the driver muxes pads 12/13 to GPIO function, sets them as outputs, drives them from the brightness callback (low = on), and forces software blink so activity shows. The OEM, by contrast, lights this LED via an in-band MCU command (`EXT_CID 0xED` / `EXT_CMD_LED_CTRL 0x17`) — the chip firmware owns the pad mux there; the mt76 approach drives the GPIOs from the host instead. See [08-front-panel-leds.md](08-front-panel-leds.md).

## Status and capabilities

Both radios run as APs concurrently under a single `wpad`/`hostapd` instance:

- **phy1 (MT7662, 5 GHz):** 802.11ac, VHT80, up to 20 dBm, factory MAC `…:04:38`, calibrated from a real factory EEPROM in flash. Confirmed connecting and passing traffic.
- **phy0 (MT7603, 2.4 GHz):** 802.11n, HT20 (e.g. channel 6 / 2437 MHz), ~19 dBm, factory-derived MAC `…:04:39`, with synthetic-EEPROM TX-power calibration. Confirmed visible, associable, and routing traffic from a client. Enumerates only on cold boot (see above).

A wireless-config caveat: multi-boot autodetection has produced duplicate `radioN` entries when the PCIe device paths differed between builds (`pci0001:` vs `1fb83000.pcie/` vs `1fb81000.pcie/`). The clean reset is `rm /etc/config/wireless && wifi config`, which regenerates exactly one radio per phy with the current kernel's device paths.

## Known issue 1 — kernel panic under sustained WiFi→LAN forwarding (rx-gro-list)

Under a sustained WiFi speedtest (~280 Mbps, several minutes), the device would **kernel-panic and reboot**, presenting as "WiFi/internet drops after a speedtest." The crash:

```
CPU 0 Unable to handle kernel paging request at virtual address 00000004
Process napi/phy1-9 ; BadVA: 00000004
Kernel panic - not syncing: Fatal exception in interrupt -> Rebooting
```

Root cause (found by `System.map` + disassembly): a NULL dereference in `__qdisc_run+0x2f8`, an `__skb_dequeue` unlink where `skb->next == NULL`:

```
lw  v1,0(s2)    # v1 = skb->next  (== NULL)
lw  v0,4(s2)    # v0 = skb->prev
sw  zero,0(s2); sw zero,4(s2)
sw  v0,4(v1)    # FAULT: store to NULL+4 = 0x4
```

The faulting path is `napi/phy1 (WiFi RX) -> napi_gro_complete -> bridge br_handle_frame_finish -> br_flood + skb_clone -> DSA __dev_queue_xmit -> __qdisc_run`. **All** the interfaces (`phy0-ap0`, `phy1-ap0`, `br-lan`, `eth0`, `lan3`) had `rx-gro-list: on` = `NETIF_F_GRO_FRAGLIST`, which mt76/OpenWrt enables globally (not a kernel default). Fraglist GRO **chains segments through `skb->next` — the same pointer the qdisc skb list uses**; forwarding and cloning those super-SKBs onto a DSA port's `fq_codel` qdisc corrupts the list. Ordinary GRO (`generic-receive-offload`) is *not* the problem; only fraglist GRO. The path (WiFi → bridge → DSA conduit) is lightly exercised, which is why it only crashes under sustained forwarding and not during normal browsing. See [04-ethernet-dsa.md](04-ethernet-dsa.md) for the DSA forward path and [04-ethernet-dsa.md](04-ethernet-dsa.md) for the related BQL/qdisc tuning.

**Fix (verified to stop the crash):**

```sh
ethtool -K phy0-ap0 rx-gro-list off
ethtool -K phy1-ap0 rx-gro-list off   # leaves generic-receive-offload on
```

This is runtime and reversible — it resets on reboot and whenever `hostapd` recreates the interface — so a permanent fix needs persistence via a `hotplug.d` script or boot hook. (As of this writing the persistence hook is not yet in the repo tree; the fix is applied at runtime.)

## Known issue 2 — MT7603 MCU timeout under long uptime

Separately from the panic above, the MT7603 can wedge after long uptime. Observed once at ~5826 s (~1 h 37 m):

```
mt7603e 0000:01:00.0: MCU message 08 (seq 11) timed out
```

The MT7603 MCU stops responding. Because a single `wpad` manages both phys, `hostapd` then blocks in uninterruptible (`D`) state waiting on the wedged radio and **stops beaconing on both bands** — so the 5 GHz radio appears to drop too, even though the MT7662 (`mt76x2e`) firmware is still running fine. The radios are physically independent (`0000:01:00.0` vs `0001:01:00.0`) and the 2.4 GHz wedge is unrelated to the VoIP PCM block (`0x1fbd0000`). The radio does **not** auto-recover: mt76 logs no reset/watchdog, and a runtime reset of the MT7603 crashes the kernel (the SCU `0x1fb000xx` runtime-write hazard), so the only recovery is a **reboot**.

Probable contributors (not the cal data): the port0 PCIe link is marginal (it had to be stabilized out of a `0x4↔0x6` flap with `PCIE_LINKDOWN_RST_EN` cleared) and may glitch occasionally, breaking MCU comms; the firmware is from 2016 (Build 20160107); thermal is possible but unconfirmed. Open follow-ups: monitor link-flap correlation, add an auto-recovery path that does not trip the runtime-SCU crash, and complete the TX-power table in the EEPROM.
